# Story Edit 迁移链审计与治理建议

更新时间：2026-04-30
范围：`apps/server/src/db/migrations/` 与 `meta/_journal.json`

## 1. 结论摘要

当前迁移链不是“编号有点乱”这么简单，而是已经出现了 **Drizzle 迁移索引与 SQL 文件目录严重脱节** 的情况。

最关键结论：

1. `meta/_journal.json` 只登记到 `0005_lame_doctor_spectrum`
2. 目录中实际存在 SQL 迁移一直到 `0038_genre_preset_project_type.sql`
3. 大量 SQL 文件没有进入 journal，属于 **orphan migrations（游离迁移）**
4. 这些 orphan 文件里还存在重复编号、前后依赖倒挂、与已登记迁移内容重叠的问题
5. 在当前状态下，**不适合上线前直接修 journal、补编号、批量重命名或强跑 `db:migrate`**
6. 目前已开始通过 `0039_*`、`0040_*`、`0041_*` 这类 append-only reconciliation migrations 落地首批治理

因此，本轮最安全的发布策略是：

- 不改历史 journal
- 不重命名历史 migration
- 不把 orphan 文件强行补进现有链
- 先冻结现状，转为单独治理任务
- 后续通过“追加 reconciliation 迁移”的方式建立可解释的 canonical 链

## 4. 关键结构性问题

### 4.1 orphan migrations：大量 SQL 根本不在 journal 中
由于 journal 只到 `0005_lame_doctor_spectrum`，所以至少以下文件都未被正式登记：

- `0005_remove_claude_openai_provider.sql`
- `0006_templates_user_enhancements.sql`
- `0007_outline_versions.sql`
- `0007_template_system_v2.sql`
- `0008_admin_role.sql`
- `0009_*`
- `0010` 到 `0038` 全部文件

这类文件即使放在目录里，也不等于会被当前 Drizzle 迁移链可靠执行。

### 4.2 已登记迁移与后续 orphan 文件内容重叠
最明显的是：

#### `0005_lame_doctor_spectrum.sql`
已经包含：

- `genre_presets` 表创建
- `outline_versions` 表创建
- `projects.genre_tag` 列
- `users.last_active_at` 列
- `project_type` 增加 `webnovel`
- 大量扩展后的 `genre` 枚举

而目录中后续又有：

- `0007_outline_versions.sql`
- `0021_genre_presets.sql`
- `0021_add_last_active_at.sql`
- `0022_project_types_expansion.sql`

这些内容与 `0005_lame_doctor_spectrum.sql` 明显存在交叉、重复或覆盖关系。

含义：

- 不能简单把后续 SQL 再加入 journal
- 否则在新库或旧库上都可能触发重复建表、重复扩枚举、重复加列、类型冲突

### 4.3 存在依赖顺序倒挂
最典型例子：

#### `0009_preferred_model.sql`
内容：

- `ALTER TABLE user_token_accounts ADD COLUMN preferred_model`

但 `user_token_accounts` 是在：

- `0032_token_system.sql`

中才被创建。

这意味着：

- `0009_preferred_model.sql` 的编号早于它依赖的表
- 若按编号顺序把它补进链，会直接失败

这不是简单的重号问题，而是 **迁移依赖拓扑已经被打乱**。

### 4.4 schema.ts 与正式 journal 已经脱节
当前 `schema.ts` 已声明大量后期结构，包括：

- token relay 相关表
- `storyNarratives`
- `settingsDeliveries`
- `editLogs`
- `userRole`
- `preferredModel`
- 更完整的 `genre` / `project_type`

但这些并没有体现在一个连续的、可重放的 Drizzle journal 中。

结果是：

- `schema.ts` 代表“目标状态”
- `journal` 代表“正式迁移历史”
- `migrations` 目录里又混杂着“部分正式历史 + 大量游离 SQL + 手工修补 SQL”

三者已经不一致。

## 5. 下一阶段验证闭环（本轮补充）

当前 reconciliation 基线已经形成，但验证闭环仍需按两条路径执行：

1. **干净库初始化验证**

   - 执行：`pnpm db:generate`、`pnpm db:verify:fresh`

2. **现有库升级路径验证**
   - 基于升级前状态演练库执行：`pnpm db:generate`、`pnpm db:verify:upgrade`
   - 对比迁移前后结构快照
   - 对 AI 配置、模板购买、Token/精灵豆余额链路做业务冒烟

本轮已新增专门操作文档：

- `docs/release-self-check/数据库验证操作手册.md`

真实目标库执行继续保持隔离，必须满足：

- 已确认连接串
- 已完成备份
- 已确认维护窗口
- 已准备回滚方案
- 已完成演练库验证

在这些前置条件满足前，不建议直接在共享环境执行 `pnpm db:migrate`。

## 6. 对 drizzle-kit migrate 的具体风险

### 风险 1：误以为目录里有 SQL 就能正常迁移
错误认知是：只要文件放在 `migrations/` 下，`drizzle-kit migrate` 就会全部顺序执行。

当前风险在于：

- journal 只认它登记过的 tag
- 目录里未登记的 orphan 文件不能被当作可靠正式链的一部分

### 风险 2：手工补 journal 可能触发重复 DDL
如果把 orphan 文件手工补进 `_journal.json`：

- 可能重复创建已有表
- 可能重复扩展已修改过的 enum
- 可能重复加列
- 可能因为已有对象与目标类型不一致而失败

### 风险 3：编号排序不能代表真实依赖顺序
如 `0009_preferred_model.sql` 依赖 `0032_token_system.sql`。

这意味着：

- 不能按编号顺序修复 journal
- 也不能仅通过重命名文件解决全部问题

### 风险 4：新环境与旧环境都会出现不同类型失败
- **新环境**：可能在重复建表/重复枚举定义处失败
- **旧环境**：可能在列已存在、类型已变化、历史数据已转换处失败

## 6. 本轮最安全的处理策略

## 推荐策略：冻结历史链，转为 append-only 治理

本轮上线前只做审计，不做破坏性治理。建议采用以下原则：

### 原则 1：不修改现有 `_journal.json`
原因：

- 当前 journal 是唯一还能代表“已知正式链”的部分
- 改错一次会让 Drizzle 对历史理解彻底失真

### 原则 2：不批量重命名旧 SQL 文件
原因：

- 编号重命名无法修复内容重叠和依赖倒挂
- 还可能让现有对照、人工记录、环境状态进一步混乱

### 原则 3：不在共享/现网数据库上尝试“补跑全部 orphan 文件”
原因：

- 风险高
- 不可逆
- 很可能在中途失败，留下半迁移状态

### 原则 4：后续采用“reconciliation 追加迁移”
正确方向应是：

1. 先确认目标环境真实数据库状态
   - `__drizzle_migrations` 表
   - 实际表结构 / enum / index / trigger
2. 以当前线上真实状态为基线
3. 对照 `schema.ts`，只为“缺少但必须存在的最终结构”创建 **新的追加迁移**
4. 将这些新迁移放在全新连续编号（例如 `0039_*` 之后）
5. 在干净数据库和升级数据库两条路径分别验证

这是当前唯一相对可控的治理路径。

## 7. 后续治理建议清单

### Phase A：只读取证
需要先收集：

1. 线上/目标数据库的 `__drizzle_migrations` 内容
2. 当前实际数据库 schema 导出
3. 与 `schema.ts` 的差异表
4. 哪些 orphan SQL 已经被手工执行过，哪些从未执行过

> 当前本地 `story_edit` 库的首轮取证结果已补充到：
> `docs/release-self-check/迁移链取证结果与对账清单.md`
> 关键事实：`__drizzle_migrations` 缺失，但大量 orphan migration 的目标结构已实际存在于数据库。

### Phase B：建立 canonical 状态表
给每个 migration 文件打标签：

- `official`：已在 journal 中
- `orphan_applied`：未在 journal 中，但环境中等效结构已存在
- `orphan_unapplied`：未在 journal 中，且环境中对应结构不存在
- `superseded`：已被后续登记 migration 覆盖，不应再直接执行
- `unsafe`：存在错误顺序或重复定义，不能直接纳入正式链

> 当前已补充首版状态矩阵：
> `docs/release-self-check/migration-文件状态矩阵.md`

### Phase C：只追加，不回写历史
输出方式建议：

- 不修旧 `000x/001x/002x/003x` 文件名
- 新建 `0039_...`、`0040_...` 等 reconciliation migrations
- 这些迁移只表达“当前环境缺什么，就补什么”

当前已完成的首批落地项：

- `0039_content_fingerprints_reconciliation.sql`
- `0040_token_numeric_reconciliation.sql`
- `0041_ai_configs_default_reconciliation.sql`

### Phase D：最终验证
至少要验证两条路径：

1. **干净数据库初始化**：能否从零构建到目标状态
2. **现有数据库升级**：能否从当前真实状态平滑到目标状态

## 8. 本轮可执行的最小发布结论

在本次上线前收口中，关于迁移链可以给出的最安全结论是：

- 已完成迁移风险识别

- 当前不建议在本轮继续改 `_journal.json`、重命名 SQL、强跑全量迁移
- 首批 append-only reconciliation migrations 已经落地，并已在本地 `story_edit` 库完成首轮执行
- 本地结构快照、migration 状态矩阵、旧体系表保留策略已同步补齐
- 仍需在真实目标数据库执行和完成双路径验证
- 应将迁移链治理拆成独立任务，按“取证 → 对账 → 追加 reconciliation migrations → 双路径验证”推进

## 10. 一句话结论

当前问题不是“把编号排整齐”就能解决，而是 **历史迁移事实、Drizzle journal、当前 schema 三者已经分叉**。上线前最安全的选择不是修历史，而是冻结历史、单独治理、追加对账迁移。
