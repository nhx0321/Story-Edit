# Story Edit 迁移链取证结果与对账清单

更新时间：2026-04-30
数据库：`story_edit`
连接方式：本地 PostgreSQL 16，用户 `story_edit`

## 1. 本次取证目标

在不修改数据库数据的前提下，确认当前数据库真实状态，回答三个问题：

1. `__drizzle_migrations` 是否存在，Drizzle 是否仍能识别历史迁移执行记录
2. 前面审计中识别的高风险 orphan / 重叠 / 倒挂迁移，数据库里到底有没有体现
3. 后续 reconciliation migration 应以什么事实为基线

## 2. 关键结论

## 结论 A：当前数据库中 **不存在** `__drizzle_migrations`
取证结果：

- `information_schema.tables` 中未发现 `public.__drizzle_migrations`
- 因此当前数据库无法从表内追溯 Drizzle 已执行历史

含义：

- 现有数据库不具备“按 Drizzle 官方迁移记录对账”的条件
- `meta/_journal.json` 只能代表代码仓库里的本地历史声明，**不能证明数据库真实执行到了哪一步**
- 后续治理必须以“实际 schema 现状”而不是“journal 推定”作为第一依据

## 结论 B：数据库实际结构已经远超 `_journal.json` 记录范围
虽然 `_journal.json` 只到 `0005_lame_doctor_spectrum`，但数据库里已经实际存在大量后续结构，包括：

- `api_channels`
- `model_pricing`
- `user_token_accounts`
- `token_consumption_logs`
- `user_api_keys`
- `token_packages`
- `token_recharge_orders`
- `user_subscriptions`
- `story_narratives`
- `settings_deliveries`
- `edit_logs`
- `genre_presets`
- `outline_versions`

说明：

- 很多后续 SQL 已经以某种方式落库
- 但这些执行事实没有被 `__drizzle_migrations` 记录，也没有被 `_journal.json` 连续表达

## 4. 取证后的新判断

### 4.1 数据库事实比代码历史更可信
因为没有 `__drizzle_migrations`，所以：

- 不能再以 `_journal.json` 推断数据库状态
- 数据库真实 schema 才是后续治理基线

### 4.2 当前不应尝试“补历史”
原因：

- 绝大多数高风险 orphan 迁移所代表的结构已经存在
- 再补历史会造成重复 DDL、类型冲突、枚举冲突、触发器重复等风险

### 4.3 后续治理应从“对账单”而不是“老 SQL 链”出发
正确姿势是：

1. 以当前数据库 schema 导出为基线
2. 以 `schema.ts` 为目标状态
3. 列出“数据库有但 schema 不需要”、“schema 需要但数据库没有”、“两边名字相同但类型/约束不同”三类差异
4. 只新增 reconciliation migrations，不回写旧历史

### 4.4 首批 reconciliation migrations 已经产出并完成本地执行
基于本次取证与后续差异对照，已落地：

- `0039_content_fingerprints_reconciliation.sql`
- `0040_token_numeric_reconciliation.sql`
- `0041_ai_configs_default_reconciliation.sql`

并已在本地 `story_edit` 库完成首轮执行：

- `0039` 以业务库用户执行成功
- `0040 / 0041` 因本地部分表属主为 `postgres`，改由本地超级用户执行成功

后续已补充的治理产物：

- 结构快照：`docs/release-self-check/db-snapshots/*`
- migration 状态矩阵：`docs/release-self-check/migration-文件状态矩阵.md`
- 旧体系表保留策略：`docs/release-self-check/旧体系表保留策略清单.md`

含义：

- 治理方向已经从“只有判断”进入“开始按差异逐项落地”
- 当前本地库已具备一套首版 reconciliation 基线
- 后续重点从识别问题转向：在真实目标数据库执行、复核、完成双路径验证

## 6. 当前阶段结论

本次取证把迁移治理从“猜测问题”推进到了“有数据库事实支撑的判断”，并进一步形成了首版本地治理闭环：

- `__drizzle_migrations` 缺失
- 真实数据库已吸收大量 orphan 迁移目标状态
- 旧历史文件无法再作为可安全重放的线性链使用
- 后续治理必须改成 **schema-first reconciliation** 模式
- 首批 `0039 / 0040 / 0041` 追加迁移已形成可执行治理链
- 本地库已完成首轮执行，并已导出结构快照、状态矩阵和旧体系表保留策略

## 7. 一句话结论
