# Story Edit 数据库验证操作手册

更新时间：2026-04-30
适用范围：`apps/server` 的 Drizzle 迁移链（含 `0039 / 0040 / 0041` reconciliation 基线）

## 1. 目标

本手册用于把当前数据库治理状态从“本地已完成 reconciliation 基线”推进到“验证路径闭合”。

本轮目标只包含：

1. 干净库初始化验证
2. 现有库升级路径验证
3. 迁移前后结构快照对比
4. 真实目标库执行前置条件固化

本手册**不授权**直接连接或操作真实目标数据库。

## 3. 环境前提

`apps/server/drizzle.config.ts` 当前通过以下方式确定目标库：

- `process.env.DATABASE_URL`
- 若未提供，则回落到本地默认库：
  - `postgresql://story_edit:story_edit_dev@localhost:5432/story_edit`

因此做任何验证前，必须先明确当前 shell 中的 `DATABASE_URL` 指向。

建议规则：

- 干净库验证：显式指向一次性演练库
- 升级路径验证：显式指向旧状态演练库
- 禁止在未确认情况下直接使用生产/共享库连接串

---

## 4. 干净库初始化验证

### 4.1 目标

验证从空数据库开始执行当前正式迁移链，是否可以完成初始化，不依赖运行时补表或手工修补。

### 4.2 操作步骤

1. 创建一套全新演练库
   - 例如：`story_edit_verify_fresh`
2. 显式设置 `DATABASE_URL`
3. 在仓库根执行：
   - `pnpm db:generate`
   - `pnpm db:verify:fresh`
4. 导出结构快照（若需完整 DDL/索引/触发器清单，需在本机补齐 `psql` 或等效工具）
5. 对照以下对象核查：
   - `content_fingerprints`
   - `ai_configs.is_default`
   - `idx_ai_configs_one_default_per_user`
   - `enforce_single_default_config` 触发器/函数
   - `model_pricing` 价格字段 bigint 口径
   - `token_consumption_logs` token 字段 bigint 口径
   - `user_sprites`
   - `sprite_bean_transactions`
   - `recharge_orders`

### 4.3 通过标准

满足以下条件视为通过：

- `db:verify:fresh` 无失败检查
- 若输出 journal 漂移告警，按“canonical 状态核查通过，但历史链不可直接重放”记录
- 不出现缺表、缺列、缺索引、缺触发器
- 不需要手动执行补丁 SQL
- AI 配置、模板购买、精灵豆、Token 主链所依赖表结构完整

---

## 5. 现有库升级路径验证

### 5.1 目标

验证一个“旧状态但可运行”的数据库，能否平滑升级到当前 reconciliation 基线。

### 5.2 演练库要求

演练库应尽量接近升级前真实状态，至少要覆盖：

- `0039` 前尚未正式接管 `content_fingerprints` 的情况
- `0040` 前 token 数值列仍为旧口径的情况
- `0041` 前 `ai_configs.is_default` 约束未正式收口的情况
- 旧 sprite / bean / shop 体系表仍保留的情况

### 5.3 操作步骤

1. 准备升级演练库
2. 备份升级前结构快照
3. 显式设置 `DATABASE_URL`
4. 在仓库根执行：
   - `pnpm db:generate`
   - `pnpm db:verify:upgrade`
5. 再次导出升级后结构快照
6. 比对升级前后差异
7. 执行业务冒烟：
   - AI 配置默认项读写
   - 模板购买
   - 精灵豆余额查询
   - 精灵豆流水写入
   - Token 余额查询
   - Token/精灵豆兑换

### 5.4 重点核查对象

- `0039_content_fingerprints_reconciliation.sql`
- `0040_token_numeric_reconciliation.sql`
- `0041_ai_configs_default_reconciliation.sql`
- `user_sprites` 是否仍可承载精灵豆余额最小链路
- `sprite_bean_transactions` 是否仍正常写入
- `recharge_orders` 是否仍支持充值统计

### 5.5 通过标准

- 升级验证脚本无失败检查
- 若输出 journal 漂移告警，视为“历史链不可直接重放”，但不单独判定本轮 canonical 对账失败
- 不出现重复索引、重复触发器、冲突约束失败
- 升级后结构与 `schema.ts`、治理文档一致
- 精灵豆与模板购买最小链路不受影响

## 6. 结构快照对比要求

每次验证至少保留两份结果：

1. 迁移前结构快照
2. 迁移后结构快照

建议至少记录：

- 所有表名
- 关键列类型
- 索引
- 触发器
- 函数
- 枚举

重点关注：

- `content_fingerprints`
- `ai_configs`
- `model_pricing`
- `token_consumption_logs`
- `user_sprites`
- `sprite_bean_transactions`
- `recharge_orders`
- 旧 sprite 候删观察组表

## 7. 与本轮精灵治理的联动要求

桌面精灵与精灵商城已停用，但以下链路仍属于现行业务，不得在验证阶段误伤：

- `spriteBean` 根路由
- `user_sprites`
- `sprite_bean_transactions`
- `recharge_orders`
- 模板购买使用精灵豆的扣减/奖励逻辑
- `settings/billing`
- `ai-config/recharge`

候删观察组仅限：

- `sprite_images`
- `sprite_items`
- `user_sprite_items`
- `sprite_conversations`
- `sprite_interaction_log`
- `sprite_text_entries`
- `sprite_ai_tasks`

本轮验证不直接删除这些表，只验证：

1. 最小链路可运行
2. 候删对象已能从代码依赖上逐步脱钩
3. 后续 append-only migration 具备落地前提

## 8. 真实目标库执行前置条件

以下动作属于高风险阶段，必须单独确认后执行：

1. 连接真实目标数据库
2. 在共享环境执行 `pnpm db:migrate`
3. 在真实库删除旧表、旧索引、旧触发器
4. 执行任何不可逆 schema 变更

执行前必须具备：

- 明确的 `DATABASE_URL`
- 维护窗口
- 可用备份
- 已验证的回滚方案
- 升级负责人
- 升级完成后的业务冒烟清单

---

## 9. 回滚预案要求

真实目标库执行前，必须提前准备：

1. 升级前完整备份
2. 结构快照
3. 数据快照或逻辑导出
4. 若迁移失败时的回滚路径：
   - 回退数据库备份
   - 恢复旧应用版本
   - 重新验证 `ai_configs`、模板购买、Token/精灵豆余额链路

若涉及未来的旧 sprite 表删除，还必须额外准备：

- 候删表行数统计
- 样本导出
- 外键与引用关系说明
- 下线后恢复步骤

---

## 10. 本轮结论

当前数据库治理已具备 reconciliation 基线，但还未完成验证闭环。

本轮收口标准是：

- 干净库初始化验证路径已定义
- 现有库升级路径验证已定义
- 真实目标库执行被明确隔离为需确认阶段
- 精灵豆最小链路被纳入数据库验证清单

在完成演练验证前，不应直接把本轮治理结论等同于“可在真实目标库安全执行”。
