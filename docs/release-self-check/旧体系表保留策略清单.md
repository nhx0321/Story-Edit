# Story Edit 旧体系表保留策略清单

更新时间：2026-04-30
范围：旧订阅 / 提现 / sprite 相关历史表
判定目标：区分哪些表仍在运行、哪些只应冻结保留、哪些进入候删观察组。

## 1. 判定原则

本清单不以“名字旧不旧”做判断，而以 3 个事实为准：

1. 当前根路由是否仍挂载对应业务入口
2. 前端是否仍有真实查询/写入调用
3. 是否仍承担现网余额、流水、管理统计或历史对账职责

额外约束：

- 财务/订单/提现类表默认优先冻结，不直接删
- 仅被未挂载旧模块引用的表，才进入“候删观察组”
- “候删”不等于立即删除，仍需先做代码清理、数据备份和双路径验证

## 2. 运行中（保留为现行表）

### 2.1 `user_sprites`
状态：`运行中`

原因：

- 虽然“旧桌面精灵陪伴系统”入口已从根路由收口，但 `user_sprites` 仍是当前精灵豆余额、经验、兑换状态的真实承载表
- `spriteBean` 仍在根路由挂载：`apps/server/src/router.ts:39`
- 服务端仍直接读写：
  - `apps/server/src/services/sprite-bean/router.ts:24`
  - `apps/server/src/services/user-account/router.ts:89`
  - `apps/server/src/services/template/router.ts:305`
- 前端仍有真实调用：
  - `apps/web/app/ai-config/recharge/page.tsx:16`
  - `apps/web/app/settings/billing/page.tsx:16`

结论：

- 该表不是历史死表，当前仍属于精灵豆/模板市场/余额兑换链路的一部分
- 本轮不得删除或脱链

### 2.2 `sprite_bean_transactions`
状态：`运行中`

原因：

- 精灵豆流水仍由该表承载

  - `apps/server/src/services/sprite/bean-service.ts:103`
  - `apps/server/src/services/sprite/bean-service.ts:201`
- 模板域仍显式依赖其交易语义：`apps/server/src/services/template/router.ts:10`

结论：

- 该表仍承担现行账务流水职责
- 应继续保留，并视为精灵豆子域的正式运行表

### 2.3 `recharge_orders`
状态：`运行中`

原因：

- 精灵豆充值订单仍由该表管理：
  - `apps/server/src/services/sprite-bean/router.ts:28`
  - `apps/server/src/services/sprite-bean/router.ts:55`

结论：

- 该表仍承担现行充值统计职责
- 暂不纳入清理范围

## 3. 冻结保留（保留数据与兼容，但不再继续扩域）

### 3.1 `subscriptions`
状态：`冻结保留`

原因：

- 但当前前端已无 `trpc.billing.*` 的真实页面调用，说明它已退出现行产品主流程

结论：

- 该表应作为历史订阅/VIP 数据保留
- 不再新增产品职责，不再作为新计费主链演进基础
- 后续只做读兼容和迁移对账，不继续扩展字段

### 3.2 `subscription_orders`

原因：

- 仅剩历史订单查看用途：`apps/server/src/services/subscription/router.ts:31`
- 当前主要价值是对账与后台历史查询，不是现行充值主链

结论：

- 不再参与当前 Token 充值链演进

### 3.3 `withdrawal_requests`
状态：`冻结保留`

原因：

- 提现实现仍完整存在：`apps/server/src/services/withdrawal/router.ts:11`

- 该表又属于财务/审核历史，不适合在未完成专项清理前直接删除

结论：

- 保留历史提现记录和审核痕迹
- 当前不作为现行业务继续扩展
- 后续若彻底下线提现体系，应先完成数据归档与后台导出能力，再考虑物理删除

## 4. 候删观察组（当前不对外运行，但需先完成专项拆除）

以下表的共同特征是：

- 主要只被旧 `spriteRouter` 或 `spriteTextRouter` 及其关联服务使用
- 这些旧路由当前都不在根路由中挂载，用户侧没有现行入口
- 但代码、静态资源、文档和部分管理逻辑还没有完全清干净，因此现在只能列为“候删”，不能直接删表

### 4.1 旧桌面精灵陪伴表

- `sprite_images`
- `sprite_items`
- `user_sprite_items`
- `sprite_conversations`
- `sprite_interaction_log`

主要证据：

- 旧路由定义：`apps/server/src/services/sprite/router.ts:34`
- 这些表的主要查询/写入集中在该旧路由内部
- 当前根路由 `apps/server/src/router.ts:21-44` 未再挂载 `sprite`
- 前端当前也未发现对应在线页面查询

当前判定：

- 进入候删观察组
- 删除前需先完成：旧 sprite 服务代码下线、静态资源依赖确认、管理后台残余能力审计

### 4.2 旧 sprite 文本生产表

主要证据：

  - `apps/server/src/services/sprite-text/service.ts`
  - `apps/server/src/services/sprite-text/ai-integration.ts`
- 根路由未挂载 `spriteText`
- 当前前端未发现在线入口

当前判定：

- 进入候删观察组
- 删除前需先确认是否还承担内部内容生产或数据回溯用途

## 5. 当前不建议直接进入“候删”的对象

### 5.1 `user_sprites`
虽然名字属于旧 sprite 域，但它已承接现行精灵豆余额和兑换状态，不应与旧陪伴系统一起误删。

### 5.2 `sprite_bean_transactions`
该表属于现行账务流水，不应按“旧 sprite 表”处理。

### 5.3 `recharge_orders`
该表仍承担充值与管理统计职责，当前应视为运行中，而不是历史残留。

## 6. 后续执行顺序建议

1. **先冻结，不删表**
   - 对 `subscriptions`、`subscription_orders`、`withdrawal_requests` 保持只读/兼容态
2. **先拆代码，再删候删表**
   - 先移除旧 `sprite` / `sprite-text` 模块的根依赖、静态资源依赖、管理入口和文档噪音
3. **做数据备份与引用复核**
   - 对候删表导出行数、样本、索引和外键关系
4. **追加治理迁移，不改旧历史链**
   - 若最终确认删除，使用新的 append-only reconciliation migration 表达，而不是回写旧迁移历史
5. **双路径验证**
   - 干净库初始化
   - 现有库升级/清理路径

## 7. 一句话结论

当前旧体系表不能一刀切处理：`user_sprites / sprite_bean_transactions / recharge_orders` 仍在运行；`subscriptions / subscription_orders / withdrawal_requests` 应冻结保留；其余旧 sprite 陪伴与文本生产表更适合进入候删观察组，等完成代码与资产拆除后再统一下线。
